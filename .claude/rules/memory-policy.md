# メモリ運用ポリシー

## 不変層原則

メモリには変化するものを書かない。許可されるのは以下の3種のみ:

1. **働き方のfeedback** — ユーザーが一度指摘したことを繰り返させないための知見。Whyを必ず添える
2. **環境の罠** — コードや公式ドキュメントから読めない非自明な事実で、再発見コストが高いもの
3. **恒久ポインタ** — 不変の参照（frontmatterは`type: reference`）。例:「タスク文脈は`notes/active-tasks.md`を見よ」

## 禁止事項

- 実装の固有名詞（関数名・ファイル名・設計の具体形）を書かない
- タスク状態・設計判断・棄却記録を書かない（すべて`~/.claude/context/<PROJECT_ID>/notes/`へ）
- CLAUDE.md / AGENTS.md / rules等に既に書かれている内容を書かない（重複は昇格済みの残骸であり将来の亡霊候補）
- `type: project`のメモリは作成しない

## ライフサイクル

- 会話で判断が覆ったら、同ターンでメモリを更新・削除する
- 安定した知見は昇格ラダーで上に送り、昇格したらメモリから消す:
  `メモリ（個人的・暫定）→ CLAUDE.md / AGENTS.md / rules（公式・安定）→ linter / hook（機械強制）`

## タスク文脈の扱い（恒久ポインタ方式）

- 各プロジェクトの`~/.claude/context/<PROJECT_ID>/notes/`に`active-tasks.md`を置く
- メモリには「タスク文脈はactive-tasks.mdを見よ」という不変の1行ポインタのみ置く
- タスクの開始・状態変化・完了時はnotes側だけを更新する

## hookによる違反検出

SessionStart hookが以下を検出し、警告をコンテキストに注入する:
- `type: project`のメモリファイルの存在
- メモリファイル数（MEMORY.mdを除く）の閾値（15件）超過

警告を受け取ったら**AskUserQuestionで修正するかどうかユーザーに確認する**こと。勝手に棚卸しを始めない。
