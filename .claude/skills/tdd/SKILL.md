# テスト駆動開発（TDD）

Kent Beck が定義したTDDの5ステップワークフロー。
設計のやりすぎも、やらなさすぎも起こりにくくなるように考慮された、プログラミング中の構造化されたワークフローである。

## 前提

傾向として「まず実装を書いてからテストを追加する」方向に流れやすい。このスキルはその傾向を明示的に上書きする。実装コードを書きたくなったら、まずテストを書く。

### TDD と混同しやすい概念

- 自動テスト: テスティングフレームワークでテストを書くこと
- 開発者テスト: 開発者自身がテストを書くこと
- テストファースト: 実装より先にテストを書くこと
- TDD: テストリスト → レッド → グリーン → リファクタ → 繰り返し、という構造化されたワークフロー全体

「テストを先にたくさん書くこと」は TDD ではない。
「設計をせずいきなりコードを書くこと」も TDD ではない。

## 5つのステップ

テストリスト(1) → テストをひとつ書く(2:RED) → テストを成功させる(3:GREEN) → リファクタリング(4:REFACTOR) → 繰り返し(5)


### Step 1: テストリスト

実装に取りかかる前に、期待される振る舞いをリストアップする。

**書くもの**: 期待される動作（振る舞い）
- 正常系はこうなる
- サービスがタイムアウトしたらこうなる
- データベースにキーがまだないときはこうなる

**書かないもの**: 実装の設計判断
- 内部のデータ構造をどうするか
- どのデザインパターンを使うか
- クラスをどう分割するか

実装設計は Step 4（リファクタリング）で扱う。テストリストの段階で混ぜ込まない。

> よくある過ち：実装の設計判断を混ぜ込んでしまうこと。まあ落ち着こう。内部実装がどうあるべきか吟味する時間は後からいくらでも確保できる。

テストリストはコード内のコメントや別ファイルとして明示的に管理する。頭の中だけで管理しない。

```
# テストリスト
- [x] 正常な入力で正しい結果を返す
- [x] 空文字列を渡すとエラーを返す
- [ ] タイムアウト時にリトライする  ← 次はこれ
- [ ] 最大リトライ回数を超えたらエラー（Step 3 で追加）
```

### Step 2: テストをひとつだけ書く（RED）

テストリストから**ひとつだけ**選び、具体的で実行可能なテストコードに翻訳する。
書いたら実行し、**失敗することを確認する**。

**ひとつだけの理由**: テストリストのすべてを先にテストコードに翻訳してしまうと、実装の過程でインターフェイスが変わったとき、投機的に書いたテストを全部書き直すことになる。1テスト→1実装の vertical slice で進める。

```
WRONG (horizontal):
  RED: test1, test2, test3, test4, test5
  GREEN: impl1, impl2, impl3, impl4, impl5

RIGHT (vertical):
  RED→GREEN: test1→impl1
  RED→GREEN: test2→impl2
  RED→GREEN: test3→impl3
```

**確認すること**:
- テストが失敗する（エラーではなく、アサーションの失敗）
- 失敗理由が「機能が未実装だから」であること（タイポやセットアップミスではない）

**やってはいけないこと**:
- 複数のテストを同時に書く
- アサーション（検証）のないテストを書く — カバレッジのためだけのテストはダブルチェックにならない

### Step 3: テストを成功させる（GREEN）

プロダクションコードを変更して、テストを成功させる。
この過程で新しいテストの必要性に気づいたら、**テストリストに追加する**（今は書かない）。

**やってはいけないこと**:
- アサーションを削除・コメントアウトして成功したふりをする — 本当に成功させる
- テスト対象の実際の出力値をコピーして期待値にペーストする — ダブルチェックにならず、妥当性確認としての価値が失われる
- このフェーズでリファクタリングを混ぜる — 「帽子を2つかぶる」ことになる。まず動かし、それから正しくする。この分離が認知負荷を下げる

### Step 4: リファクタリング（REFACTOR）

テストが通った状態で、コードを整理する。**ここで初めて実装設計の判断を行う。**

#### ふたつの設計

設計には2種類ある：

1. **インターフェイス設計**: 振る舞いはどのように呼び出されるべきか — Step 2（テストを書く段階）で扱う
2. **実装設計**: システムはその振る舞いをどう実装すべきか — Step 4（リファクタリング段階）で扱う

この2つを同時にやらない。分離することで、一度にひとつの関心事だけに集中できる。

#### リファクタリングの注意点

- **重複は「指令」ではなく「ヒント」**: 重複を見つけたら自動的に除去するのではなく、設計改善のきっかけとして捉える。早すぎる抽象化に注意
- **やりすぎない**: コードの整理整頓は気持ちがいいが、次のテストに向かうのが怖いからといって過剰にリファクタリングしない
- **テストを壊さない**: リファクタリング中、すべてのテストがグリーンのままであることを都度確認する

### Step 5: 繰り返し

テストリストが空になるまで — あるいはコードの動作に対する不安が退屈に変わるまで — Step 2 に戻る。

## サイクルの粒度

各サイクルは小さく保つ。テストひとつ → 数行の実装 → 小さなリファクタリング、が理想的なリズム。大きなステップを踏もうとしたら、もっと小さく分割できないか考える。

## テストの品質

| 性質 | 良いテスト | 悪いテスト |
|------|-----------|-----------|
| **最小限** | ひとつの振る舞い。名前に「and」が入ったら分割する | `test_validates_email_and_domain_and_whitespace` |
| **明確** | 名前が振る舞いを記述する | `test_1`, `test_it_works` |
| **意図を示す** | 望ましい API を示す | 何をテストしているか不明 |
| **実コード** | モックは本当に不可避な場合のみ | すべてをモックで置き換える |
| **リファクタ耐性** | 内部実装を変えてもテストが壊れない | リファクタリングするとテストが壊れるが、振る舞いは変わっていない |

**モックのアンチパターン**: モックの追加やテストユーティリティの作成時は [references/anti-patterns.md](references/anti-patterns.md) を参照する。

## ワークフロー外の注意

### テストより先にプロダクションコードを書いてしまったら

TDD ワークフローの中では、テストが先。先にプロダクションコードを書いた場合、そのコードを「参考」として残すとテストが既存コードの追認になりやすい。TDD の効果（まだ存在しない振る舞いを定義すること）を得るには、先に書いたコードを手放して TDD サイクルで書き直す方が確実。

探索のために書いたプロトタイプがある場合も、TDD サイクルで書き直す。

### バグ修正

バグを見つけたら、まずそのバグを再現する失敗テストを書く。テストが失敗することを確認してから修正する。こうすることで、修正の証明と回帰テストの両方が得られる。

### 行き詰まったとき

| 問題 | 対処 |
|------|------|
| テストの書き方がわからない | 望ましい API を先に書く。アサーションから書き始める |
| テストが複雑すぎる | 設計が複雑すぎるサイン。インターフェイスを単純化する |
| 何もかもモックが必要 | コードの結合度が高すぎる。依存性注入を検討する |
| テストのセットアップが巨大 | ヘルパーを抽出する。それでも複雑なら設計を見直す |

## 完了前チェックリスト

作業完了を宣言する前に確認する：

- [ ] すべての新しい振る舞いに対応するテストがある
- [ ] 各テストが失敗するのを確認してから実装した
- [ ] 各テストの失敗理由が「機能未実装」だった（タイポやセットアップミスではない）
- [ ] 各テストに対して最小限のコードで成功させた
- [ ] すべてのテストがパスしている
- [ ] テスト出力にエラーや警告がない
- [ ] テストリストが空になっている（または残項目を意図的に先送りした理由がある）

## 中断・再開時の注意

作業を中断する場合は、テストリストの状態（どこまで完了したか、途中で追加された項目）を保存すること。再開時はテストリストを確認し、次にどのテストを書くべきか把握してから再開する。
