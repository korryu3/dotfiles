# セッション保存

セッションの作業・判断・残タスクを、人間がレビュー・編集できる引き継ぎドキュメントとして書き出す。

## 原則: このドキュメントだけで会話の全てがわかるようにする

読者は元の会話を一切見られない。このドキュメントを読むだけで「何が起きて、なぜそうなって、次に何をすべきか」が全て理解できなければならない。説明だけでは伝えきれないコンテキストがある場合は「次のセッションで読むべきファイル」で実際のコードを指定する。

ただし会話の逐語的なトレースではなく、引き継ぎに必要な情報を蒸留して書く。

書くときは常に自問する：**会話を見ていない人がこれを読んで、作業を正しく引き継げるか？**

一見冗長な表現でも、書かなくて理解できないよりはマシである。

- 固有名詞・技術用語の初出時に、それが何かを説明する
- 判断を書くときは、選択肢の中身と比較の根拠も書く
- 議論を書くときは、結論だけでなく発端と経緯も書く
- 「あの」「例の」「さっきの」のような会話内参照は使わない
- 選択肢・質問・回答を要約するときは、件数だけでなく内容も書く（「4項目を解消」ではなく各項目と回答を記載する）
- 状態が未確定のときは「可能性あり」で終わらせず、次のセッションで確認すべき手順を書く

## ドキュメントテンプレート

セッション内容に応じて不要なセクションは省略してよい。

```markdown
---
created_at: YYYY-MM-DD HH:MM
session_topic: （一言で何のセッションだったか）
---

# セッション引き継ぎ

## 前回のセッション
このセッションが前回の引き継ぎから再開された場合に記載する。
- 前回のドキュメント: `~/.claude/context/<PROJECT_ID>/handoffs/YYYY-MM-DD/xxx.md`
- 前回からの要約:（今回の作業を理解するのに必要な範囲で、前回までの経緯を要約する）

## ゴール
ユーザーの元々のリクエストを正確に記録する。

## 完了した作業
- 何をやって、どうなったか
- 変更したファイルがあればパスを記載

## 未完了の作業
- まだ終わっていないこと
- 次のセッションで最初にやるべきことを明確に
- 複数ある場合は優先順位と理由を記載する
- 全て完了している場合は「なし（このプロジェクトは完結）」と明記する

## 重要な判断とその理由
「何を選んだか」「なぜそれを選んだか」「他の選択肢は何だったか」を書く。

## 議論の経緯
ユーザーとの重要な議論を時系列で記録する。
結論だけでなく、どういう流れでその結論に至ったかを残す。

## 関連ファイル
読み書きした主要なファイルと役割・変更内容の一覧。
- `path/to/file` — 何をしたか

## 次のセッションで読むべきファイル
このドキュメントだけでは伝えきれないコンテキストを持つファイル。本当に必要なものだけに絞る。
- `path/to/file` — 理由

## 発見・注意点
ハマりどころ、制約、前提条件など、コードやgit historyからは読み取りにくい情報。

## 現在の状態
- ブランチ名、直近のコミット
- テスト・リントの状態
- 動作確認の状況
- 未確定の状態がある場合は、次のセッションで確認すべきコマンドを記載する（例: `git log --merges -1`）

## 進捗

| タスク | 状態 |
|---|---|
| ... | 完了 / 進行中 / 未着手 |
```

## 保存先

`~/.claude/context/<PROJECT_ID>/handoffs/YYYY-MM-DD/<トピック>.md`

PROJECT_ID は `~/.claude/scripts/project-id.sh` を実行して取得する（例: `/Users/foo/bar` → `-Users-foo-bar`）。取得できない場合は `default` を使用。

## 手順

```
- [ ] セッション全体を振り返り、テンプレートに沿って引き継ぎドキュメントを作成
- [ ] `~/.claude/context/<PROJECT_ID>/handoffs/YYYY-MM-DD/` ディレクトリがなければ作成
- [ ] ドキュメントを保存
- [ ] ユーザーに内容を提示して「修正したい箇所はありますか？」と確認
- [ ] コピペ用コマンドを表示
```

ユーザーが編集を指示したら反映する。

## コピペ用コマンド

保存後、次のセッションでそのまま貼り付けて使えるコマンドを表示する：

```
/resuming-session ~/.claude/context/<PROJECT_ID>/handoffs/YYYY-MM-DD/<トピック>.md
```

実際のファイルパスを埋めること。
