# コメントレビュー

原則:
- **コードの修正は一切行わない。レポートのみを出力する。**
- **差分の追加・変更された行のコメントのみが対象。**
- **迷ったら「確認依頼」に分類する。**
- **コメント単体で判断しない。** 必ず Read ツールで前後のコードを確認した上で判定する。
- **コメントの内容は、最小限の文章に最大限の情報を詰め込むことを目指す。**

## 1. 対象ファイルの特定

### 引数なし（デフォルト）: 現在のブランチの変更

まずベースブランチを以下の優先順で特定する:

1. `gh pr view --json baseRefName --jq .baseRefName` を実行し、PRが存在すればそのベースブランチを使う
2. PRが存在しない場合、AskUserQuestion ツールを使って「ベースブランチは develop ですか？違う場合は指定してください」とユーザーに確認する

ベースブランチが特定できたら `git diff <base-branch>...HEAD --name-only` を実行して変更ファイル一覧を取得する。

### `pr <PR番号>` が指定された場合

`gh pr diff $1 --name-only` を実行して変更ファイル一覧を取得する。

### 除外対象

以下に該当するファイルはレビュー対象から除外する:
- 自動生成ファイル（`gen/` ディレクトリ配下、`*.generated.*` など）
- ロックファイル（`yarn.lock`, `package-lock.json` など）

## 2. 差分の取得とコメントの抽出

対象ファイルの差分を取得する。

ただし以下はレビュー対象外とする:
- ツール指示コメント（`@ts-ignore`, `@ts-expect-error`, `eslint-disable`, `type: ignore`, `noqa` など）
- shebang（`#!/...`）やマジックコメント（`# -*- coding: ... -*-` など）

コメントを含む行を特定したら、Read ツールで対象ファイルを読み、**HEAD時点のファイルにおける正確な行番号**を確認する。

## 3. コメントの分類と判定

コメントのベストプラクティスに関する知識をフルに活用して判定すること。以下はこのプロジェクト固有の原則と分類の指針であり、網羅的なチェックリストではない。

### 削除候補

読者に新しい情報を与えないコメント。コードを読めば得られる情報を繰り返しているだけのもの。

### 改善提案

コメントとして存在する意義はあるが、内容や形式に改善の余地があるもの。

このプロジェクト固有のルール:
- **TODO/NOTE には背景（Why）が必要。** 「何をすべきか」だけでなく「なぜ現状こうなっているか」が初見のエンジニアにも伝わること
- **コメントは日本語で書く。** ライブラリ由来や技術用語が英語の方が自然な場合は除く
- **コードと矛盾するコメントは修正が必要**

TODO/NOTE のプレフィックス:

| プレフィックス | 用途 |
|---|---|
| `NOTE` | 現状のコードがそうなっている背景・理由を説明する |
| `TODO` | 今後必要な対応を記述する |
| `FIXME` | 既知の不具合で、修正が必要な箇所 |
| `HACK` | 理想的でない一時的な回避策 |

**例:**
- 悪い: `// TODO: あとで直す` → 背景も対応方針も不明
- 良い: `// NOTE: 現在はデータ量が少ないため問題ないが、N+1問題が発生している` + `// TODO: JOIN を使った取得メソッドに置き換える`

### 確認依頼

判断に迷うもの。報告するが、削除・修正の提案はしない。判断をユーザーに委ねる。
コメントアウトされたコードは意図的に残している可能性があるため、常にこのカテゴリに分類する。

## 4. レポートの出力

以下のテンプレートで出力する。該当セクションがない場合は省略する。
連番はセクションをまたいで通しで振る。カテゴリ別にグルーピングし、注目すべきものを上に配置する（改善提案 → 確認依頼 → 削除候補の順）。

```markdown
## コメントレビュー結果

### サマリー
- ブランチ: `<branch-name>` (<base-branch> からの差分)
- 🗑️ 削除候補: N件 / ✏️ 改善提案: N件 / 💬 確認依頼: N件

### ✏️ 改善提案
1. `path/to/file.tsx` L78: `// TODO: あとで直す` — 背景と対応方針が不明確
   - **改善後**:
     - `// NOTE: ...背景...`
     - `// TODO: ...対応方針...`

### 💬 確認依頼
2. `path/to/file.tsx` L95: コメントアウトされたコード（3行）。意図的に残していますか？

### 🗑️ 削除候補
3. `path/to/file.tsx` L42: `// ユーザーを取得する` — 直後の `getUser()` を読めばわかる

### 次のアクション
- **削除候補**(3)は全て削除します。残したいものがあれば番号を指定してください
- **改善提案**(1)を適用します。不要なものや修正したい内容があれば番号と指示を添えてください
- **確認依頼**(2)について判断をお願いします: 削除 / 残す
```
