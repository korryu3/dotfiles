# PR理解度確認テスト

「読んだ」と「理解した」は違う。このスキルはPRの内容をユーザーが仕組みレベルで理解しているかを検証する。

## ワークフロー

```
テスト進捗:
- [ ] Step 1: PRを読み込み、出題領域を決める
- [ ] Step 2: 3問出題する
- [ ] Step 3: 採点・フィードバック
- [ ] Step 4: 質疑応答
- [ ] Step 5: 次ラウンドまたは総合評価
```

## Step 1: PRを読み込み、出題領域を決める

PRのURLまたはブランチ情報がなければユーザーに聞く。以下を取得する:

- PRの差分全体（`gh pr diff`または`git diff`）
- PRのdescription、関連issue
- コミット履歴

取得した内容から、出題する知識領域を決める:

- **必須**: 仕組み（どう動くか）、変更の全体像（概念レベルで何が変わったか）
- **重要**: 意図・背景（なぜこの変更が必要か）
- **状況次第**: 影響範囲、エッジケース、設計判断

PRの性質に応じて「状況次第」を格上げする。例: 破壊的変更があれば影響範囲は必須。

## Step 2: 3問出題する

1ラウンド3問、自由記述形式。コードの丸暗記ではなく仕組みの理解を問う。最初は「必須」領域から出し、正答できたら徐々に深い領域へ進む。

## Step 3: 採点・フィードバック

各問題に `✅ 理解OK` / `⚠️ 部分的` / `❌ 要復習` を付ける。

採点の姿勢は**厳しめだが建設的**。目的は「知らないといけないことを知らなかったら指摘する」こと。

- 正解: 簡潔に認める。過剰に褒めない
- 部分的に正解: 合っている部分を認めた上で、抜けを具体的に指摘
- 不正解: すぐ正解を教えず、まず何を見ればわかるかヒントを出す
- 「知るべきことを知らなかった」場合は明確にそう伝える

## Step 4: 質疑応答

「質問はありますか？なければ次に進みます」と促す。質問があれば丁寧に答える。ここは「テスト」ではなく「学習」のフェーズ。

## Step 5: 次ラウンドまたは総合評価

**次ラウンドへ**: 弱かった領域を深掘り、または全問正解なら深い領域へ進む。

**終了条件**: 全領域カバー済み / ユーザーが終了を希望 / PRの読み直しを推奨すべき理解不足。

**総合評価テンプレート**:

```
## 総合評価

### 理解度: B（概ね理解しているが、一部重要な抜けあり）

**よく理解できていた点:**
- ...

**要復習:**
- ...

**次のアクション:**
- ...
```

S（実装者と同等以上）/ A（正しく理解、細部に若干の抜け）/ B（概ね理解、重要な抜け1-2箇所）/ C（表面的理解、核心に誤解）/ D（読み直し推奨）

## 注意事項

- PRの規模が大きい場合、最も重要な変更に絞ってテストする
- ユーザーが「ここは知らなくていい」と言った領域は出題しない
